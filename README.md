## About this project

This is a **complete, AI-generated training program** designed to take a working TypeScript developer from zero C# knowledge to "productive on a real .NET codebase" in one to two weeks. The entire curriculum — 31 lessons, quizzes, runnable code samples, lesson ordering, and every TypeScript → C# analogy — was designed and written by [Claude](https://claude.com/claude-code), Anthropic's AI assistant, working from a brief that specified the target audience (junior fullstack TS dev), timeline (1–2 weeks), and goal (fundamental C#/.NET fluency, not surface syntax).

**The program's core teaching strategy:** every single lesson opens with an explicit mapping from something you already know in TypeScript to the C# equivalent. `Promise<T>` becomes `Task<T>`. `Array.map` becomes `.Select`. `type User = { ... }` becomes `public sealed record User`. No concept is introduced in a vacuum — it's always anchored to a TS idiom you've written a thousand times, so the new syntax lands on an existing mental model instead of an empty one.

**The delivery:** a **Blazor WebAssembly** app. The entire .NET runtime, the curriculum, and the [Roslyn](https://learn.microsoft.com/en-us/dotnet/csharp/roslyn-sdk/) compiler itself all run inside your browser as WebAssembly. When you write C# and hit **Run**, the code is compiled by a real `Microsoft.CodeAnalysis.CSharp.CSharpCompilation` instance, emitted to a memory stream, reflected into a live `Assembly`, and invoked — all without a backend. You can host the whole thing as static files on GitHub Pages. No server, no sandbox, no external code-execution API. Just you and the .NET runtime.

> **First-load download is ~20 MB** (the .NET WASM runtime plus Roslyn plus the reference assembly set). It's cached after that, so subsequent visits are instant. Compilation of a typical lesson snippet takes ~100–400 ms once the runtime is warm.

## Who this is for

- **TypeScript / Node developers** who need to read, write, or contribute to a C# / .NET codebase and don't have time for a traditional months-long course
- **Bootcamp grads and junior engineers** whose first language was JS/TS and who are now on a polyglot team
- **Anyone curious** about modern .NET who wants a fast, honest, opinionated tour of the parts that matter in real codebases (as opposed to textbook C# from 2015)

If you've never written TypeScript, this program isn't tuned for you — pick a more general intro. If you *have*, the analogies will do a lot of the heavy lifting for you.

## Quick start

You need the **.NET 10 SDK** installed. Everything else is self-contained.

```bash
cd CSharpLearningLab
dotnet run
```

Then open [http://localhost:5080](http://localhost:5080) in your browser. Progress is saved to `localStorage`, so you can quit and pick up where you left off.

```bash
# Verify your SDK
dotnet --version
# 10.0.201  ← or newer
```

## How it works

Three moving parts, all inside your browser tab:

| Layer | Tech | What it does |
|---|---|---|
| **UI** | Vanilla HTML/CSS/JS in `wwwroot/` | Renders the sidebar, lesson content, code editor, and progress bar. No framework, no build step. |
| **Runtime bridge** | Blazor WebAssembly | Boots the .NET WASM runtime. Exposes three `[JSInvokable]` methods (`ListLessons`, `GetLesson`, `RunCode`) that the vanilla JS calls via `DotNet.invokeMethodAsync`. |
| **Compiler** | `Microsoft.CodeAnalysis.CSharp` + [`Basic.Reference.Assemblies.Net90`](https://www.nuget.org/packages/Basic.Reference.Assemblies.Net90) | Builds a `CSharpCompilation`, emits a PE stream, loads it via `Assembly.Load(byte[])`, and invokes the entry point with `Console.Out` redirected into a `StringWriter`. All in-process, in-browser. |

When you hit **Run**, the flow is:

```
 Browser JS                          .NET WebAssembly                    Roslyn
     │                                      │                              │
     │  DotNet.invokeMethodAsync            │                              │
     │  ("CSharpLearningLab","RunCode",…)   │                              │
     ├─────────────────────────────────────▶│                              │
     │                                      │  ParseText → Compilation     │
     │                                      ├─────────────────────────────▶│
     │                                      │  Emit(PE stream) → Diagnostics│
     │                                      │◀─────────────────────────────│
     │                                      │                              │
     │                                      │  Assembly.Load(bytes)        │
     │                                      │  entryPoint.Invoke(null)     │
     │                                      │  capture Console.Out         │
     │                                      │                              │
     │      JSON {output, error, ms}        │                              │
     │◀─────────────────────────────────────│                              │
```

Every lesson is a plain C# record in [`Data/LessonContent.cs`](Data/LessonContent.cs) — no database, no CMS. The entire curriculum lives in one file, in code, under version control.

### Notable footguns I hit along the way

A few WASM-specific gotchas that make this more fiddly than a server-hosted Roslyn runner:

- **`CSharpScript.RunAsync` doesn't work.** It internally calls `MetadataReference.CreateFromFile(assembly.Location)` on the host assembly, but every WASM assembly has an empty `Location`. Solution: use the lower-level `CSharpCompilation.Create` → `Emit` → `Assembly.Load(byte[])` path.
- **No disk-resident reference assemblies.** You can't point the compiler at `System.Private.CoreLib.dll` on the filesystem. [`Basic.Reference.Assemblies.Net90`](https://www.nuget.org/packages/Basic.Reference.Assemblies.Net90) embeds the full .NET 9 ref set as byte arrays inside the NuGet package, so Roslyn has everything it needs in-memory.
- **Async top-level statements generate a blocking wrapper.** The C# compiler wraps async `<Main>$` in a sync `Main` that calls `.GetAwaiter().GetResult()`. That blocks the single-threaded WASM runtime with `"Cannot wait on monitors on this runtime."` Solution: skip `Assembly.EntryPoint`, hunt for `<Main>$` by name via reflection, and `await` it directly.
- **Top-level statements must precede type declarations.** Several lesson snippets had `public class Foo { ... }` followed by `var x = new Foo();` — legal under `CSharpScript`, illegal in regular C# source mode (CS8803). Solution: the code runner parses each snippet, walks the `CompilationUnitSyntax.Members`, buckets `GlobalStatementSyntax` into one pile and type declarations into another, and re-emits the source with statements first.

## Tech stack

- **.NET 10** / C# 14
- **Blazor WebAssembly** — boots the .NET runtime inside your browser
- **`Microsoft.CodeAnalysis.CSharp`** — a real Roslyn compiler running in WASM

## Add your own lessons

Every lesson is a C# record in [`Data/LessonContent.cs`](Data/LessonContent.cs). Adding one is as simple as appending a new `Lesson(...)` to the `All` array:

```csharp
new Lesson(
    Id: "32-your-topic",
    Order: 32,
    Title: "Your New Topic",
    Category: "Intermediate",
    TsAnalogy: "TS: Array.flatMap → C#: .SelectMany",
    Explanation: "Here's how it works...",
    StarterCode: "Console.WriteLine(\"hello from lesson 32\");",
    SampleSolution: "",
    ExpectedOutput: "hello from lesson 32\n",
    KeyPoints: ["Point 1", "Point 2"],
    Quizzes:
    [
        new Quiz(
            Question: "What does SelectMany do?",
            Choices: ["Maps", "Filters", "Flat-maps", "Reduces"],
            CorrectIndex: 2,
            Explanation: "It projects and flattens."
        )
    ]
)
```

Rebuild and reload. Lessons show up in the sidebar in `Order`, grouped by `Category`.

## A note on AI-generated curriculum

Because the entire program was written by an LLM, a few things are worth being upfront about:

- **It's opinionated by design.** The lessons are tuned for *one* audience (TS developers becoming useful on a real .NET codebase) and deliberately skip things that don't serve that goal. There is no lesson on WinForms, no deep dive on unsafe pointers, no LINQ query syntax — not because they don't exist, but because a junior TS dev joining a modern .NET team in 2026 almost certainly won't hit them.
- **Run the code.** Every starter snippet was smoke-tested against the actual Roslyn runner before shipping, but LLMs make mistakes. If you find a bug in a lesson, open an issue or a PR — I'll take both.
- **The analogies are approximations.** `Task<T>` is *like* `Promise<T>`, but the two have meaningful differences around cold-vs-hot and exception propagation. The lessons prioritize getting you productive, then flag the subtleties so you know where the edges are.

If you find a concept that was explained poorly, or a TS → C# analogy that clicks better than mine, please open a PR. The entire curriculum is in [one file](Data/LessonContent.cs) and is meant to be edited.

## Security note

Because the Blazor WebAssembly port moves every layer into the browser, the security story is actually **much nicer** than the original server-hosted version:

- **The code runs inside the visitor's own browser tab.** Nothing is sent back to a server. If a visitor writes hostile code, it can only affect their own tab — not other visitors, not your infrastructure.
- **The WebAssembly sandbox is enforced by the browser.** User snippets can't touch the filesystem, open network sockets, or escape the tab.
- **The runner still enforces a 5-second per-execution timeout** so a `while (true) { }` freezes one tab, not the world.

In other words: this version is safe to deploy publicly. The original minimal-API version on `git log master` was not.
